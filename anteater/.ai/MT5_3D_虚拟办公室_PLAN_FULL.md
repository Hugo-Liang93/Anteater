# MT5 3D 虚拟办公室项目实施计划（PLAN_FULL）

## 1. 项目目标

### 1.1 背景
当前系统已经具备 MT5 后端服务、多模块职责拆分，以及前端 React + TypeScript + Vite 的基础形态。项目希望进一步把原本偏“后台面板”的展示方式，升级为一个可视化、可交互、具备动画风格的 **3D 虚拟办公室 / 数字交易工作室**。

该工作室不是单纯装饰性 3D 页面，而是用于将 MT5 交易系统中的核心模块人格化、场景化、流程化，让使用者可以直接看到系统当前在做什么、流程卡在哪里、哪个模块异常、哪些模块正在协作。

### 1.2 核心目标
- 将 MT5 后端主链路模块映射为多个“员工角色”
- 通过 3D 场景可视化系统实时工作状态
- 通过动画、光效、状态标识展现模块运行情况
- 保留现有 React 后台式 UI 的效率优势
- 支持后续继续扩展为策略实验室、监控中心、回测可视化等能力

### 1.3 首版成功标准
1. 页面中出现可运行的 3D 工作室主场景
2. 至少 6 个核心角色与 MT5 主链路模块建立映射
3. 角色状态可以根据 mock 数据或后端数据实时变化
4. 点击角色可以查看当前任务、状态、关键指标
5. 页面具备顶部总览、底部事件流、右侧详情面板
6. 从“行情 → 指标 → 信号 → 风控 → 执行”的主链路可以被直观看见

## 2. 设计原则

### 2.1 先功能，后美术
优先完成系统状态可视化、模块映射、数据驱动和交互流程，再逐步升级角色模型、材质风格和复杂动画。

### 2.2 先主链路，后扩展区
首版只覆盖 MT5 实盘主链路的核心流程，不同时展开所有二级能力。

### 2.3 先占位角色，后正式角色
首版允许使用低成本几何体占位、简化人形、基础状态动画。等前后端协议跑通后，再引入统一美术角色。

### 2.4 3D 场景与业务 UI 分层
- 3D 层：负责空间、角色、光效、链路表现
- 2D UI 层：负责详情、事件流、告警、总览信息
- 通信与状态层：负责连接后端、映射数据、驱动表现

### 2.5 后端状态不直接耦合前端表现
前端不要直接依赖后端内部类和实现细节，而应通过统一协议映射为前端的 `StudioAgent`、`StudioEvent` 等统一结构。

## 3. 总体方案

### 3.1 产品定位
项目定位为：

**MT5 3D 虚拟交易工作室**

它既是一个业务前端，也是一个系统运行状态可视化界面。
它不只是展示“界面”，而是展示“系统此刻正在工作”。

### 3.2 总体架构分层
1. 后端模块层：MT5 接入、行情采集、数据缓存/持久化、指标计算、信号生成、风控审核、下单执行、健康检查/监控、FastAPI 对外服务、后续回测/实验
2. 通信层：WebSocket、REST API、状态聚合接口、事件流接口
3. 前端状态层：全局 store、角色状态、事件流状态、选中角色、告警与总览状态
4. 3D 场景层：办公室主场景、区域分区、员工角色、状态动画、数据流特效、异常警示
5. 2D UI 层：TopBar、Sidebar、角色详情面板、底部事件流、总览信息卡、告警区域
6. 资源层：角色模型、动作资源、场景道具、材质与贴图、图标与 UI 视觉素材

---

## 4. 技术选型
- 前端基础：React / TypeScript / Vite
- 3D 场景：three / @react-three/fiber / @react-three/drei
- 状态管理：zustand
- UI 动效：framer-motion
- 样式：Tailwind CSS（如当前项目已使用则沿用）
- 通信：WebSocket 为主，REST API 为辅
- 资源格式与工具：GLB / glTF、Blender、Mixamo、Spline / 模型查看器

### 选型理由
这套方案兼顾当前已有 React 技术栈、后续业务 UI 扩展性，以及 3D 场景与普通 React 页面在同一工程协作的需求，适合从“后台”平滑升级到“3D 业务可视化前端”。

## 5. 角色划分方案

### 5.1 首版核心角色（MVP）
1. **MT5 接入员**：MT5 连接、账户状态、终端状态；重点关注连接正常、重连、断线
2. **行情采集员**：Tick / Quote / K线采集；重点关注当前 symbol、tick 速率、最新报价时间
3. **指标工程师**：EMA / SMA / RSI / ATR 等指标计算；重点关注指标计算状态、最新值、耗时
4. **信号策略员**：信号模块、策略判定；重点关注 BUY / SELL / HOLD、策略名、置信度
5. **风控官**：止损、冷却、仓位、风控审核；重点关注通过 / 拒绝、拒绝原因、冷却时间
6. **执行员**：订单发送、修改、平仓、执行结果；重点关注订单状态、执行耗时、成交/拒单结果

### 5.2 二期扩展角色
- 数据管理员（缓存/持久化）
- 巡检员（健康检查/监控）
- API 接待员（FastAPI 请求入口）
- 回测实验员（参数优化/实验室）

## 6. 场景布局方案

### 6.1 首版场景定位
首版使用“俯视微斜角”的固定办公室视角，而非自由漫游，优先保障观察效率与开发速度。

### 6.2 功能区域划分
- 左侧：接入区（MT5 接入员、通讯控制台、连接状态灯）
- 左中：行情区（行情采集员、数据入口装置、发光数据流）
- 中央：分析区（指标工程师、图表屏幕、指标面板）
- 中右：决策区（信号策略员、决策白板、BUY/SELL/HOLD 标识）
- 右侧：风控区（风控官、风险闸门、审核灯）
- 右下：执行区（执行员、订单终端、成功/失败发光反馈）

### 6.3 二期区域
- 数据仓库区
- 巡检区
- API 前台区
- 策略实验室

## 7. 页面结构方案

### 7.1 主页面结构
- TopBar（顶部总览栏）
- Sidebar（左侧导航/辅助区域）
- Studio3D（3D 主场景）
- EmployeeDetail（右侧详情面板）
- BottomEventFeed（底部事件流）

### 7.2 UI 分层
- 第一层：3D 场景（角色、区域、链路、光效、异常状态）
- 第二层：角色浮层（角色名、当前状态、当前任务）
- 第三层：详情面板（所选角色的详细运行信息）
- 第四层：全局总览（系统健康度、账户、symbol、告警数等）
- 第五层：事件流（主链路事件和模块变化）

## 8. 数据协议设计

### 8.1 角色状态结构
```ts
export type AgentStatus =
  | "idle"
  | "working"
  | "walking"
  | "thinking"
  | "waiting"
  | "warning"
  | "error"
  | "success"
  | "disconnected"
  | "reconnecting"
  | "reviewing"
  | "blocked"
  | "approved"
  | "submitting"
  | "executed"
  | "rejected"
  | "judging"
  | "signal_ready";

export interface StudioAgent {
  id: string;
  name: string;
  module: string;
  zone: string;
  status: AgentStatus;
  task: string;
  symbol?: string;
  metrics?: Record<string, string | number | boolean>;
  alertLevel?: "none" | "info" | "warning" | "error";
  updatedAt: string;
}
```

### 8.2 事件流结构
```ts
export interface StudioEvent {
  eventId: string;
  type: string;
  source: string;
  target?: string;
  symbol?: string;
  level: "info" | "warning" | "error";
  message: string;
  createdAt: string;
}
```

### 8.3 映射原则
后端内部状态 → 统一协议 → 前端 store → UI / 场景渲染

## 10. 分阶段实施计划

### 阶段 0：准备阶段
确认现有项目前端基础、依赖、目录和主页面入口。

### 阶段 1：3D 办公室骨架
搭建 Canvas 场景入口、相机、灯光、地板、6 个主要功能区域、6 个员工占位对象。

### 阶段 2：角色状态驱动
定义 `StudioAgent` 类型与角色状态机，完成 mock 数据驱动、状态颜色变化、图标变化、简单动画。

### 阶段 3：2D UI 与 3D 联动
点击角色打开详情面板，悬停显示简要状态，顶部总览显示账户 / 健康度 / 告警数，底部事件流滚动展示系统事件。

### 阶段 4：主链路流转可视化
绘制主链路连接线，实现数据流粒子或发光点，不同类型流转使用不同颜色/光效，关键事件触发局部演出。

### 阶段 5：接入真实后端数据
设计后端聚合状态接口，接入 WebSocket 与 REST，完成后端状态到前端协议的 mapper，异常与事件统一落到事件流。

### 阶段 6：美术升级
确定统一角色风格规范，制作或选取统一底模，输出岗位变体，替换占位角色，并可选接入 Mixamo / Blender 动作。

### 阶段 7：扩展能力
巡检员、API 接待员、回测实验室、多 symbol 切换、场景聚焦与告警历史筛选等。

## 12. 风险点与应对
- 过早陷入美术细节：先用简易占位角色验证功能与数据链路
- 后端状态碎片化：后端提供统一聚合状态，不让前端直接拼装内部对象
- 3D 场景与业务 UI 相互污染：严格按分层管理
- 一次做太大：严格分阶段，每阶段必须有可见成果
- 过早上正式模型：先跑通逻辑，再替换正式模型

---

## 13. 开发优先级建议
- P0：技术栈落地、3D 场景入口、主场景骨架、6 个角色占位、前端状态结构、mock 数据驱动
- P1：点击交互、详情面板、顶部总览、底部事件流、主链路连线
- P2：WebSocket 对接、后端真实状态接入、事件映射、告警表现
- P3：模型替换、美术升级、动作升级、特效增强
- P4：实验室 / API 前台 / 扩展区域

---

## 14. 建议的首周执行清单
- Day 1：安装 3D 相关依赖，创建 `Studio3D` 场景入口，跑通最小 Canvas
- Day 2：增加地面、灯光、相机，搭建 6 个区域占位
- Day 3：增加 6 个员工占位角色，加入基础名字标签
- Day 4：定义 `StudioAgent`、`StudioEvent`，创建 zustand store，接入 mock 数据
- Day 5：根据状态改变角色颜色 / 图标 / 简单动画
- Day 6：接入右侧详情面板，完成点击选中联动
- Day 7：接入底部事件流，增加主链路连线

首周结束预期成果：一个可演示的“功能型 MT5 3D 办公室雏形”。

---

## 15. 最终建议
本项目最重要的不是一开始做多精致，而是做出一个真正能承载 MT5 业务状态的“活的办公室”。

建议顺序：
1. 先做功能办公室
2. 再做链路办公室
3. 最后做美术办公室

换句话说：
- 先让员工“有工作”
- 再让员工“会协作”
- 最后再让员工“更好看”
