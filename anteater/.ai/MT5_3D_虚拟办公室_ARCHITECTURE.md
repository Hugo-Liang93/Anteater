# MT5 3D 虚拟办公室前端架构说明（ARCHITECTURE）

## 1. 文档目的

本文档用于说明 MT5 3D 虚拟办公室项目的技术架构、模块边界、数据流转方式与前后端协作方式。  
目标是让后续开发、扩展、联调与维护都有统一的技术理解。

---

## 2. 项目定位

该项目是一个 **React + TypeScript + 3D 场景** 的业务前端，用于把 MT5 交易系统的后端模块可视化为一个可交互、可观察、具备动画表达能力的“虚拟办公室”。

它不是纯管理后台，也不是纯游戏，而是：

- 以业务状态为核心
- 以 3D 场景为表达层
- 以 React UI 为信息层
- 以前后端实时通信为驱动层

---

## 3. 分层架构

整体建议采用 6 层结构：

### 3.1 资源层
负责静态资源：
- 角色模型（GLB）
- 动作资源
- 场景道具
- 图标与贴图
- 主题颜色与视觉配置

### 3.2 3D 表现层
负责：
- Canvas 场景
- 相机
- 灯光
- 区域布局
- 角色模型/占位对象
- 流程连线
- 粒子与光效
- 状态动画

### 3.3 UI 表现层
负责：
- TopBar
- Sidebar
- 右侧详情面板
- 底部事件流
- 全局告警
- 浮层与 tooltip

### 3.4 状态层
负责：
- 角色状态 store
- 事件流 store
- UI 状态 store
- 选中角色
- 总览统计

### 3.5 数据适配层
负责：
- WebSocket 消息解析
- REST 响应解析
- 后端原始字段到前端统一协议映射
- 默认值补全
- 枚举归一化

### 3.6 后端接口层
负责与 MT5 后端服务交互：
- WebSocket
- REST API
- 状态聚合接口
- 历史事件接口
- 健康状态接口

## 6. 核心模块说明

## 6.1 Studio3D
`Studio3D` 是 3D 办公室的顶层入口组件。  
职责：

- 创建 Canvas
- 接入主场景
- 连接相机、灯光、区域、角色与流转线
- 不负责业务状态生成，只负责消费状态

### 不应承担的职责
- 不直接写后端请求逻辑
- 不直接拼装后端返回结构
- 不承载复杂 UI 逻辑

---

## 6.2 StudioScene
负责办公室空间组织：
- 场景根容器
- 区域位置组织
- 角色摆位
- 主链路连线

它应尽量保持“声明式场景布局”特征。

---

## 6.3 AgentAvatar
负责渲染单个角色：
- 位置
- 状态表现
- hover / selected 状态
- 模型或占位几何体
- 局部动画

### 输入
- `StudioAgent`
- 是否选中
- 是否悬停
- 视觉配置

---

## 6.4 Overlay UI
包括：
- `TopBar`
- `EmployeeDetail`
- `BottomEventFeed`
- `GlobalStatus`

它们从 store 中读取状态，不直接依赖 3D 对象实例。

---

## 6.5 Store
推荐拆为：

### agent.store
负责：
- `agents`
- `selectedAgentId`
- `setAgents`
- `updateAgent`
- `selectAgent`

### event.store
负责：
- `events`
- `appendEvent`
- `trimEvents`
- `clearEvents`

### ui.store
负责：
- 详情面板显隐
- 当前场景模式
- 过滤器
- 告警展示开关

---

## 7. 数据流转原则

## 7.1 单向流转
建议始终保持：

**后端消息 → mapper → store → UI/Scene**

而不要让场景层自己去“猜”业务状态。

---

## 7.2 协议统一
前端只认识统一协议对象，例如：
- `StudioAgent`
- `StudioEvent`

这样即使后端字段未来变化，也只需要改 mapper 层。

---

## 7.3 mock 与 live 一致
mock 数据必须尽量与未来 live 数据结构一致。  
这样前期开发不会白做。

---

## 8. 通信架构

## 8.1 WebSocket
适用于实时状态：
- 模块状态变化
- 告警
- 新事件
- 连接断开/恢复

### 建议职责
- 单独 client 维护连接
- handlers 分发不同消息类型
- store 内只接收统一协议对象

## 8.2 REST API
适用于：
- 首屏初始化数据
- 历史事件
- 某角色详情补充数据
- 非高频统计数据

---

## 9. 状态同步策略

## 9.1 首屏加载
流程建议：
1. 加载基础页面
2. 请求初始角色状态
3. 请求最近事件
4. 建立 WebSocket
5. 进入实时同步

## 9.2 实时更新
- 收到 agent update → 更新对应角色
- 收到 new event → 追加事件流
- 收到 summary update → 更新顶部总览

## 9.3 断线策略
- 顶部显示连接异常
- MT5 接入员进入 `disconnected/reconnecting`
- WebSocket 自动重连
- 恢复后拉取一次增量或全量状态

---

## 10. UI 与 3D 的职责边界

## 10.1 3D 负责“看见”
- 谁在哪
- 谁正在工作
- 流程走向
- 哪里有异常
- 状态氛围

## 10.2 UI 负责“看懂”
- 当前任务到底是什么
- metrics 详情
- 事件消息
- 错误原因
- 精确时间与数据

## 10.3 一条原则
不要把大量文字塞进 3D 场景。  
3D 负责“感知”，UI 负责“解释”。

---

## 11. 场景坐标与布局建议

建议采用固定坐标布局，而不是动态自动布局。  
首版推荐每个区域与角色都有明确坐标配置，例如：

- mt5_zone: `[-8, 0, 0]`
- market_zone: `[-4, 0, 0]`
- indicator_zone: `[0, 0, 0]`
- signal_zone: `[4, 0, 0]`
- risk_zone: `[8, 0, 0]`
- execution_zone: `[12, 0, -2]`

### 原则
- 流程从左到右
- 观察逻辑直观
- 后续可以微调，不应一开始做自动布局系统

---

## 12. 性能策略

## 12.1 首版性能目标
首版目标不是极致性能，而是可维护与稳定。  
但仍需避免明显浪费。

## 12.2 建议
- 少量角色时避免过度优化
- 粒子数量先控制
- 事件列表做长度截断
- 避免每帧做重计算
- 复杂 derived state 放到 store selector 或 memo 中

## 12.3 正式模型接入后
- 压缩 GLB
- 控制贴图尺寸
- 避免过多实时阴影
- 必要时对道具做简化

---

## 13. 测试建议

## 13.1 前端逻辑测试
优先测：
- mapper
- store
- 状态切换逻辑
- 事件追加逻辑

## 13.2 场景测试
关注：
- 是否渲染成功
- 是否能点击角色
- 状态变化是否有视觉反馈

## 13.3 联调测试
关注：
- WebSocket 连接与重连
- 消息容错
- 后端数据异常时前端是否降级

---

## 14. 错误处理策略

## 14.1 mapper 容错
- 缺字段时补默认值
- 非法枚举映射到 `warning` 或 `idle`
- 记录异常日志但不让页面崩溃

## 14.2 UI 降级
- 没有选中角色时显示空状态
- 没有事件时显示空态
- 连接异常时显示横幅提醒

## 14.3 场景降级
- 模型加载失败时回退到占位几何体
- 某角色缺数据时仍保留其基础位置信息

---

## 15. 扩展架构建议

后续可按模块继续扩展：
- `lab/`：实验室页面
- `monitor/`：监控页面
- `api-zone/`：API 前台
- `storage-zone/`：数据仓库区

但建议保持同样架构原则：
**通信 → mapper → store → 表现层**

---

## 16. 结论

本项目架构的核心不是“做一个炫酷 3D 页面”，而是建立一套可持续扩展的业务可视化前端体系。

最关键的三条原则：

1. **3D 场景只负责表现，不直接承担业务拼装**
2. **前端只消费统一协议，不直接依赖后端内部结构**
3. **UI 与 3D 分工明确：一个负责解释，一个负责呈现**
